## <br />

### [Ai] 엣지 케이스 발굴용 변경 영향도 AI 리포트 PoC - 1

대규모 코드베이스 프로젝트 운영 중 기능 및 VOC 개선으로 인한 사이드 이펙트·엣지 케이스를 놓칠 수 있는 문제를 AI로 해결하는 방법을 탐색중이다.

### 문제 도출 과정

#### 1) 사람인 채용센터 유지보수 및 운영

- 대규모/복잡 도메인 비즈니스 로직 코드 변경 시 side effect 탐지 난이도 높음

- 기획 변경 이력/테스트 자산 분산으로 QA 누락 가능성 상존

- 코드 리뷰를 자율적으로 거치고 있으나 개발 시점으로부터 장기간 경과에 따른 컨텍스트 손실

- 운영 중 개선이 필요한 비즈니스 로직을 발견해도, 사이드 이펙트 우려로 인해 리팩토링 불가

#### 2) TFT 프로젝트 QA

- 런칭 전 QA 중 기획·디자인 변경이 발생한 경우, 기 통과된 관련 기능의 N차 QA가 보장되지 않기에 `엣지 케이스`를 개발자 개인이 능동적으로 도출, 검증해야 함

#### 문제 도출 : 엣지 케이스 도출의 개인 의존성

- 영향 범위 분석이 표준화되어 있지 않으므로 엣지 케이스 누락 가능성 존재

- 테스트 범위 산정이 개인 역량과 프로젝트 일정에 좌우됨

- 테스트 자산 재사용 불가

→ QA 효율 저하, 깨진유리창 / VOC 인입에 영향을줌

<br />

### 해결 방향

기능 변경 MR 생성시 영향 범위/핵심 시나리오/엣지 케이스를 AI를 활용하여 회귀 위험 선제 대응

#### 주요 진행 과정

1. AI QA 운영 방식 조사 — 변경 영향도 AI 리포트(PR 게이트) 선택

2. develop 브랜치/특정 커밋(Optional) 기준 diff를 추출해 LLM에 전달, 영향 영역(impactedAreas), 핵심 시나리오(coreScenarios), 사이드 이펙트(sideEffects), 엣지 케이스(edgeCases)를 JSON+마크다운으로 생성하는 스크립트 작성

3. CI/CD 연동 — MR 생성 시 자동 실행

4. hiring의 일부 페이지 정책이 담긴 applicant-manage/README.md 내용을 프롬프트에 청크 분할 주입하는 방식 구현 (컨텍스트 오버로드 방지)
