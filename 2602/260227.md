# Today I Learned

## [Ai] 운영 서비스 정책 컨텍스트화 및 LLM 활용 방안 PoC - 1

### 아이디어 도출

[엣지 케이스 발굴용 AI 리포트 PoC](260225.md#ai-엣지-케이스-발굴용-변경-영향도-ai-리포트-poc---1)를 진행해보니 AI가 코드 변경의 영향도를 분석하고 테스트 케이스를 제안하려면, 서비스 도메인을 먼저 이해시켜야 한다는걸 느꼈다.
이전 방식은 diff만 보고 리포트를 생성하기 때문에, 서비스의 구체적인 정책 맥락 없이 일반적인 수준의 엣지 케이스만 생성할 수 있기 때문이다.

```markdown
// 명세 없이는 LLM이 추론하기 어려운 정책들

- A 기능의 n번째 단계 모달에서 뒤로 가기 시 어떤 단계의 데이터가 보존되고 어떤 데이터가 초기화되는지
- 후불형 서비스에서 특정 단계에 추가로 제한되는 기능 목록
- 마감 90일 경과 공고에서 차단되는 액션의 범위
- 면접 요청 플로우에서 어떤 모달이 순차적으로 열리는지
- 메일 템플릿 확인 이후 실제 메일이 발송됨
```

따라서 런칭 전 수행되었던 12,000개의 TC로부터 LLM이 읽을 수 있는 서비스 정책 컨텍스트를 생성하고,
이를 기반으로 AI가 정확한 영향도 분석과 엣지 케이스 발굴을 수행할 수 있도록 하는 선행 작업이 필요하다는 생각이 들었다.

정책 컨텍스트가 마련되면 장기적으로는 기획 → 디자인 → FE/BE 개발 → QA → 배포 → 모니터링까지 전 운영 과정을 자동화하는 멀티 에이전트 시스템의 지식 기반이 될 수 있지 않을까 🤔

### 문제

#### 2-1. LLM의 도메인 지식 부재

- LLM은 코드 diff를 읽고 일반적인 엣지 케이스를 추론할 수 있지만, 서비스 고유의 정책은 알 수 없다

#### 2-2. TC는 있지만 LLM이 읽을 수 없는 형태

- 공통, 공고, 후보자, 관리자, 메일/알림톡, ...등의 탭으로 구조화되어있는 약 12,000개의 TC는 스프레드시트 구조로 되어 있어 LLM에 직접 주입하기 부적합하다. 또한 각 TC를 그대로 주입하면 약 68만 토큰을 소모하게 된다. (= 품질 저하 & 비용 낭비)

- 또한 TC는 조건-기대결과의 나열들이라 정책 규칙으로 추상화되어 있지 않으며, E2E 시나리오의 조각들이다.

#### 2-3. 기존 정책 문서와 코드 간 불일치

- TC는 런칭 시점이기 때문에 이후 7개월간 업데이트된 기획 문서(Axure/Figma), 추가 및 삭제된 기능/디자인, 실제 코드 구현 사이에 정책 드리프트(Policy Drift) 가 발생한다. AI가 잘못된 정책을 학습하면 오히려 노이즈가 증가한다.

### 해결 방향

#### 3-1. TC → 서비스 정책 컨텍스트 변환 파이프라인

```
Google Sheets TC (탭별, 기능별 CSV .policies/raw/...에 내보내기)
    ↓ generate-policy.js
서비스 정책 컨텍스트 문서 (.policy.md)
    ↓ verify-policy.js
소스 코드와 대조
    ↓ generate-policy.js 오류 수정 + 누락 조건 보완 + 재검토
LLM 프롬프트 주입
    ↓ ai-impact-report.js (diff와 연관있는 기능 관련 프롬프트만 조건부 로드)
MR생성 OR 스크립트 구동시 더 정확한 영향도 분석 + 엣지 케이스 생성
```

#### 3-2. 파일 경로 기반 조건부 컨텍스트 주입

diff에서 변경된 파일 경로를 보고, 관련 정책 문서만 선택적으로 LLM에 주입한다.
관련 없는 변경에서는 정책 문서를 주입하지 않아 불필요한 토큰 낭비를 방지함

```js
const POLICY_MAP = [
  { match: /applicant-manage/, file: ".policies/applicant-manage.policy.md" },
  { match: /applicant-view/, file: ".policies/applicant-view.policy.md" },
  { match: /recruit/, file: ".policies/recruit.policy.md" },
  //...
];
```

#### 3-3. Jira 결함 이력 연동

TC의 Jira 컬럼(DEFECT-xxx)을 정책 문서에 포함시켜, 과거에 실제로 발생한 결함 패턴을 LLM이 참조할 수 있게 한다.

- 중요도 S + Jira 있음 → ⚠️ DEFECT-xxx 강조 표시 → AI가 동일 패턴 감지시 high risk로 판단
- 중요도 B + Jira 있음 → (DEFECT-xxx) → 단순 누락 이력
