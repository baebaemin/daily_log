## Today I Learned

### [git] merge commit revert하기

- 상황 : 새로 개발 중이던 프로젝트가 홀딩되어 다시 채용센터 운영으로 돌아왔다. 정기 배포 스케줄에 맞춰 수정 건을 develop에 병합해두었는데 다른 업무의 선배포가 필요했던 상황이라 revert를 하게 되었다.

- 시도 : develop -> main으로 가는 배포 MR에는 여러 커밋들이 있었는데, 그중 내 작업 커밋을 골라 `git revert {commit-hash}`를 실행시켰지만 실패했다.

- 원인 : 해당 {commit-hash}가 feature branch에서 작업했던 내 커밋 내역들이 묶여있는 병합 커밋이었기 때문이다.

- 해결 : `git revert -m 1 {commit-hash}` 명령어 입력

- 해설 : merge commit은 2개의 부모 커밋을 가지고 있다.
  - 부모 1 : 원래 브랜치 (develop)
  - 부모 2 : 병합된 브랜치 (feature/xxx...)

  ```
  develop:  A -- B -- C -----E----- M (merge commit)
                        \         /
  feature/xxx:            D --- F
  ```

  - M의 입장에서 부모 1은 E(develop에 병합 전 상태)이고, 부모 2는 F이다(feature/xxx의 마지막 커밋)
  - `git revert -m 1 {M}`는 부모 1을 mainline으로 유지하고, 병합으로 추가된 내용을 제거한다는 의미다. 즉, D와 F가 develop으로부터 제거되고 E로 되돌린다.

- 추가 : 만약 (드물지만) `git revert -m 2 {M}`를 한다면...?

  ```
  develop:  A -- B -- C --E(사라짐)-- M (merge commit)
                        \         /
  feature/xxx:            D --- F
  ```

  - feature/xxx의 D,F는 유지가 되고 develop의 추가 작업(E)가 제거된다.

- 추추가 : `-m 1 옵션`을 모르고 cli로 실행하려다 배포 30분 앞두고 revert에 자꾸 실패해서 gitlab UI로 revert하도록 도와주셨다ㅎㅎ..
  - repository > History > commit > Options > Revert
