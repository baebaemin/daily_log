### [CSS] `:global()` descendant selector로 컴포넌트의 내부 element 스타일링

- 문제: 디자인 시스템의 TextInput 사용 중 예외적으로 input 값을 가운데 정렬해야했다. DS 컴포넌트가 `className` prop을 받아주긴 하지만 root element 한정이라서, 내부 `input` 태그에는 외부에서 className을 꽂아넣을 방법이 없었다. DS submodule 수정 및 버전업을 바로 하진 못하는 상황.

- 시도: 처음엔 그냥 wrapper로 한 번 더 감싸서 wrapper에 스타일을 주면 되지 않을까 싶었는데, input의 `text-align`은 실제 input 태그에 직접 닿아야 하는 속성이라 wrapper 레벨에서는 적용되지 않았다.

- 해결: DS 컴포넌트를 감싸는 outer wrapper에 modular className을 하나 주고, 그 scope 안에서 `:global(input)` descendant selector로 plain tag를 직접 타게팅했다. modular class scope 안에서 사용하면 적용 범위도 내 scope 안으로만 한정시킬 수 있다.

  ```scss
  .step1__input {
    :global(input) {
      text-align: center;
    }
  }
  ```

### [개발 프로세스] Spike — 시간 제한된 사전 검증 작업

- speckit과 figma mcp를 통해 구현 task를 받아보다 보면 "DS Stepper의 props가 이런 동작을 지원할 것이다", "Chip에 closable variant가 있을 것이다"... 같은 가정에 기반해 task를 잘게 쪼개진 항목이 있다. (특히, ctx가 무거워지지 않기 위해 새로운 세션으로 열었을 때) 문제는 implement 수행시 앞서 승인한 task graph가 통째로 흔들리는 경우가 있어 결국 토큰 소비량이 늘어나게 된다.

- XP/Agile에서는 이런 가정을 본격적인 작업 착수 전에 미리 확인/깨보는 작업을 `Spike`라고 부른다고 한다. 문서나 소스를 제한된 시간 내에 들여다보고 "지원함 / 지원 안 함 / 대안 X 필요" 같은 결론만 확정짓고 빠져나오는 것이다. 

- 그래서 요즘은 기능 선개발 후 DS로 구성된 디자인을 입혀야하면, 구현 전 앞 단계에 "spike: DS props 검증" 같은 항목을 별도로 두기 시작했다. 본 task에 들어가기 전에 기술 리스크를 사전 제거했더니 중간에 specs와 구현 코드 등의 재귀적 수정 등이 무너지는 일이 줄었다.
