## Today I Learned

### [Spec-Driven Development] 기능만 완성된 화면 위에 figma 디자인 완성본 입히기

- 세 번째 TFT 참여로 ... 런칭 기한을 맞추기 위해 Confluence PRD + 기획자님이 주신 prototype을 바탕으로 spec-driven-development(이하 SDD)를 speckit으로 한참 진행하던 중이었다. 도중에 claude code의 agents view 기능이 발표되면서, 많게는 5개의 세션을 병렬로 띄워놓고 페이지별로 빠르게 개발해나갈 수 있었다. 페이지마다 worktree로 작업 공간은 분리하되, PRD/specs 같은 산출물은 통합 브랜치에 모아서 실시간으로 업데이트하는 식으로 관리했다. 코드의 퀄리티는 FE개발팀원 모두가 참여한 harness submodule 덕에 보장될 수 있었다. 이제 디자인이 완료되어 SDD로 만들어둔 기능 위주의 wireframe 위에 figma mcp를 연동해 디자인 시스템과 공통 컴포넌트를 입히는 단계에 들어가게 됐다.

- 문제는 디자인이 단순 스타일 변경이 아니라 기존 기능 요구사항 자체를 바꾼다는 점이었다. (e.g. 한 페이지로 끝나던 등록 절차가 URL 4개로 쪼개져 STEP별 화면이 됨) 그리고 그동안 쌓아둔 specs / claude artifacts / 구현 코드의 양이 이미 방대해서, figma 링크 던지고 디자인 반영해주라고 던지면 ctx도 터지고 DS 적용도 제대로 안 될 것 같았다 ... (code connect 도입을 미처 못 한 상황)

- 그래서 디자인 적용 작업을 위한 프롬프트를 다음 단계로 쪼개고, 각 단계 사이마다 내가 인간 토큰으로 ^ㅡ^ 승인하는 흐름으로 진행했다. 

  1. conflict review : 기존 spec과 figma 디자인 사이의 requirement conflict를 먼저 검토 요청. 스타일 차이가 아니라 기능 흐름이 달라지는 지점을 도출.
    a. conflict decision : review 결과를 내가 확인하고 어떻게 정리할지 결정/답변.
  2. spec amendment : 확정된 decision을 기반으로 기존 speckit spec을 개정. 단, `plan.md`/`tasks.md`는 직접 수정하면 안 됨. 수정·폐기 필요 항목만 별도의 파일로 추가.
  3. re-plan : 승인된 amendment 기준으로 구현 plan/tasks를 overwrite.
  4. implement : re-plan이 추천하는 순서대로, 승인된 amendment와 re-plan에 포함된 변경사항만 구현. 각 task 시작 전에 관련 design-system mapping을 먼저 확인.

- 산출물이 이미 두꺼운 상태에서 디자인을 새로 얹는 작업이라 부담이 있다. 기존 구현코드 분석 / 정책 정합성 파악 / 사내 디자인 시스템과 공통 훅 활용 / figma mcp 연결 등 토큰이 몇 배로 빨리 사라짐... 가장 좋은 워크플로우는 기획+디자인이 다 나온 상태에서 깔끔하게 시작하는 것이겠지만 기획+BE부터 나오는대로 급하게 해야했기에 ㅜ ㅜ 아쉽다.
