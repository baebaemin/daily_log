## Today I Learned

### [Ai] 엣지 케이스 TC 중심 QA 게이트 프롬프트 보완

[AI가 생성하는 TC](/2603/260318.md#ai-엣지-케이스-tc-중심-qa-게이트-poc) 품질을 개선하면서 추가/수정한 프롬프트 규칙 로그.

- 규칙이 늘어날수록 프롬프트가 무거워지는 트레이드오프가 있다..
- 정밀할수록 좋긴 한데 diff가 크거나 회귀 정도에 따라 비효율적일수도 있어서 어느 시점엔 규칙 병합이나 우선순위 정리가 필요할 것 같다 🤔

| 날짜  | 카테고리         | 규칙                                                                                                               | 개선점                                                                                                                                                                                      |
| ----- | ---------------- | ------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 03/18 | QA writing rules | `Keep each TC atomic` / `generate a separate TC per flow`                                                          | 두 가지의 TC를 하나로 합치는 것 방지 (e.g., 메일 템플릿에서 면접 유형 값은 면접 일정 요청, 장소 값은 면접 일정 등록 과정에서 확인해야 함)                                                   |
|       |                  | `Do not combine expected results from different flows`                                                             | 서로 다른 모달/템플릿 변수 결과를 하나로 묶는 것 금지                                                                                                                                       |
|       |                  | `Write every TC from the end-user's perspective` / `verify the observable UI result`                               | 사용자 관점으로 작성 (e.g., "place 변수가 컨트롤러에서 전달되는지" → "면접 일정 등록 시 비대면(Google Meet) 타입 선택 후 메일 보내기 모달에서 장소 항목에 'Google Meet'가 표시되는지 확인") |
|       |                  | `Do not generalize a TC to "all controllers" or "all components"`                                                  | "모든 컴포넌트" 또는 "모든 컨트롤러"로 일반화하지 않기                                                                                                                                      |
|       |                  | 변경되지 않은 코드라도 변경된 코드와 data/logic dependency가 있으면 TC 추가 허용                                   | 논리적 연결 없이 단순히 같은 파일/함수에 있는 코드 기반 TC는 금지                                                                                                                           |
| 03/19 | Priority rules   | `P0 only for release-blocking regressions` / `P1 for important regressions but do not fully block release` / `...` | 메인 시나리오가 아닌 접근성 관련 테스트는 P0 미만의 중요도를 갖도록 수정                                                                                                                    |
|       |                  | `remove cases that overlap with existing edgeCases in user action, constraint, and expected result.`               | Edge Case랑 Priority별 TC 중복 제한                                                                                                                                                         |
|       | QA writing rules | `For combobox/listbox a11y, do not require listbox focus transfer when aria-activedescendant keeps focus`          | 콤보박스 & 옵션 리스트에서 초점 이동 자체를 성공 기준이 아닌 보조공학이 리스트를 인식하고 탐색 가능한지를 기준으로 작성하도록 수정                                                          |
| 03/20 | QA writing rules | `Do not generate manual UI QA TCs that require intentionally altering API responses ...`                           | API 응답 등을 일부러 비정상적으로 바꿔야만 검증 가능한 시나리오 생성 금지                                                                                                                   |
|       |                  | `Describe state-related TCs in user action ...`                                                                    | 복구/원복 같은 상태 표현 대신 사용자 행동 순서로 작성 (e.g., "B로 복구했을때" → "B 선택했다가 A 선택했다가 다시 B 선택했을 때")                                                             |
|       |                  | fn:`isInvalidNonEditableMailVariableQaTask`                                                                        | 메일 편집기/메일 본문/템플릿 변수 문맥에서 항목을 사용자가 직접 edit할 수 있다고 전제하지 않기                                                                                              |
|       |                  | `do not combine expectations across those surfaces ...`                                                            | 서로 다른 surface의 기대 결과를 한 TC에 합치지 않기                                                                                                                                         |
|       | general          | `QA_CHECKLIST_MIN_ITEMS` 14 → 10                                                                                   | 체크리스트 개수 압박 줄이기, 근거가 약할 경우 TC에서 제외                                                                                                                                   |
| 03/23 | Priority rules   | 명시적 명세/요구사항이 없고 사용자 기대 UX에 가까운 TC는 `P0~P2` 대신 `UX`로 분류 - 카테고리 추가                  | e.g., 오른쪽 버튼 클릭 시 탭 리스트가 왼쪽으로 부드럽게 이동하는지 확인 → 별도의 애니메이션 명세가 없음                                                                                     |
|       | general          | 공용 컴포넌트 수정 시 각 사용처별 TC 수행 용이하도록 scope가 1개라면 체크리스트에 함께 기재                        |                                                                                                                                                                                             |
