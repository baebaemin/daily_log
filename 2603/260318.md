## Today I Learned

### [Ai] 엣지 케이스 TC 중심 QA 게이트 PoC

- 지난 번 수행했던 [변경점 기준 AI 리포트 생성](../2602/260225.md#ai-엣지-케이스-발굴용-변경-영향도-ai-리포트-poc---1) 작업의 산출물을 활용해 내부 QA를 위한 TC 생성 gate PoC를 진행했다.
- QA 리소스 부족이나 소규모 변경으로 내부 QA만 진행할 때 발생하는 운영 이슈를 보완하고, 릴리즈 리스크 부담 없이 리팩토링·최적화 등 코드 개선을 가능하게 하는 것이 목적이다. (사이드 이펙트가 무서워서 눈 감은 곳들이 있음 😑)

#### 구조

- MR diff 분석 기반 Gemini API 연동으로 핵심 시나리오·엣지 케이스·사이드 이펙트 리포트를 생성한다.
- 운영 건 개발 -> MR 생성 -> `qa-gate` 뱃지 달아둔 MR만 diff 기반으로 AI 리포트 생성 -> 해당 artifacts 기반으로 엣지 케이스 기준 TC 생성 -> TC 수행 -> 통과시 merge 허용 ... 의 구조이다
- 체크리스트는 우선순위별(P0/P1/P2/EDGE/UX) 자동 생성되여 GitLab MR·CI와 동기화한다.

#### 동작 방식

```
MR 오픈 (qa-gate 라벨 부착)

  └─▶ [qa-gate-sync] ai-impact-report.js
        └─ git diff 수집 → Gemini API 호출 → 영향도 리포트(JSON) 생성
        └─ gitlab-qa-gate-checklist.js syn
              └─ 리포트 기반 체크리스트 생성 → GitLab MR description 동기화

  └─▶ [qa-gate-verify] gitlab-qa-gate-checklist.js verify
        └─ 체크리스트 완료 여부 검증
```

<br />

#### ai-impact-report.js로 생성되는 리포트 구성 항목

| 항목            | 내용                                                 |
| --------------- | ---------------------------------------------------- |
| 영향 범위       | 변경으로 인한 위험 영역 (riskLevel: HIGH/MEDIUM/LOW) |
| 핵심 시나리오   | 검증해야 할 주요 사용자 흐름                         |
| 사이드 이펙트   | 간접적으로 영향받을 수 있는 영역                     |
| 엣지 케이스     | 리스크 가설 + 구체적 QA 체크 항목 (2~3개)            |
| 테스트 우선순위 | P0/P1/P2/UX 매트릭스 + mustRun 여부                  |

<br />

#### gitlab-qa-gate-checklist.js로 생성되는 체크리스트

- P0 → P1 → P2 → EDGE -> UX 순으로 우선순위 그룹화

- 최소 항목 수 보장 (기본 14건, QA_CHECKLIST_MIN_ITEMS로 조정 가능) (=> 억지 케이스를 만드는 것 같아서 제거)

- 잘못 생성된 QA 항목은 모니터링 하면서 필터링 되도록 프롬프트 수정 중 (도달 불가 UI 상태, 존재하지 않는 필터 라벨 등)

<br />

#### 파일들

| 파일                                      | 역할                                      |
| ----------------------------------------- | ----------------------------------------- |
| `.scripts/ai-impact-report.js`            | diff 분석 + Gemini API 호출 + 리포트 생성 |
| `.scripts/gitlab-qa-gate-checklist.js`    | 체크리스트 생성 / MR 동기화 / 검증        |
| `.gitlab/ci/qa-gate-hiring.gitlab-ci.yml` | CI 파이프라인 정의 (sync + verify job)    |

<br />

#### 사용 방법

- MR 생성 후 qa-gate 라벨 부착

- CI 파이프라인 자동 실행 — MR description에 체크리스트 자동 삽입

- 팀원이 항목 검토 후 체크 완료 -> qa-gate-verify 통과

#### PoC 후기

- 현재까지 약 8건의 운영·유지보수 MR에 적용해 본 결과, 대규모 diff(약 84k자 이상)에서는 TC가 평균 16개, 소규모 diff(7k자 미만)에서는 7~8개가 생성되었고, 이를 바탕으로 총 약 100건의 테스트를 수행했다. 그중 실제 결함을 발견한 TC는 3건이었으며, 3%이지만 충분히 유의미하다고 판단되어 팀내 qa-gate 문화를 공유할 예정이다.

- 그러나 특정 상황에서는 AI가 부적절한 TC를 내기도 하므로, 프롬프트·검증 로직을 지속적으로 보완하며 사용중이다. 특히 서비스 정책·도메인 맥락 없이 변경 diff만으로 판단하기 때문에, 아래와 같은 이슈가 있었다.
  - 메일 템플릿 모달에서 본 내용이 실제 발송 메일 본문에도 동일하게 반영되는지까지 묶어 검증해야 하는데, 그 연결을 TC에 잘 담지 못함
  - 공용 컴포넌트가 바뀐 경우 사용처(화면·플로우)별로 TC를 나누어 써야 하는데, 한 곳에만 국한되거나 지나치게 일반화됨
  - 애니메이션 등 명세에 없는 UX 기대를 TC에 포함하는 등, 검증 범위가 불필요하게 넓어짐

- 따라서 점진적으로 명세 의존도를 높이는 방향으로 가야할 것 같고 ... 초반에 먼저 해소하고 싶은 한계는 LLM이 diff만 보고 TC를 도출하는 것에서 오는 맥락 부족이기 때문에 Jira API로 관련 ITS를 연동하려 한다. MR과 연결된 이슈에서 기대 결과, 기획·운영 의도, 정책 조건을 가져올 수 있다면 명세 추론의 정확도를 높이고 TC scope을 좁히는 데 도움이 될 것 같다!

- ITS 본문을 그대로 프롬프트 컨텍스트에 넣어 TC 품질 개선

  ```zsh
  MR 생성
    - 브랜치/커밋에서 ITS 번호 추출
    - Jira API로 이슈 본문(=초안 명세) 가져옴
    - speckit이 ITS 본문을 해석해서 구조화된 명세 생성
    - 생성된 명세 + diff → TC 생성
    - MR 체크리스트 반영
  ```

- speckit은 원래 사람이 쓴 명세를 정제하는 도구인데, ITS 본문으로부터 spec.md 초안을 생성하도록 할 예정이다.

  ```node
  [기존 speckit 흐름]
  사람이 spec.md 작성 -> clarify -> plan -> tasks -> analyze

  [ITS 연동 흐름]
  Jira API로 ITS 본문 가져옴
  -> LLM이 ITS 본문을 해석해서 spec.md 초안 생성
  -> speckit.clarify로 모호한 부분 탐지
  -> 구조화된 spec + diff
  -> TC 생성
  ```
