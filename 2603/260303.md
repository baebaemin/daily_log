## [Ai] 운영 서비스 정책 컨텍스트화 및 LLM 활용 방안 PoC - 2

### 생성

서비스의 구체적인 정책 맥락을 생성하기 위해 CSV 파일로 내보낸 TC(우선 약 180건)를 기반으로 컨텍스트 문서 초안을 ai로 생성해보았다.

### 구현

- `.scripts/generate-policy.js`
  - `.policies/*.policy.md`를 생성하는 역할.
  - TC에서 아래의 항목들을 추출하고 정책 언어로 요약하도록 한다.
    - 플로우
    - 상태 전이
    - 엣지 케이스
    - 권한 분기
    - 결함 이력

4. 중간 영역에서 필터 및 설정 기능의 노출 상태를 확인한다.

# 2. 상태 전이 정책

- **노출 뷰 유지:** 사용자가 리스트뷰 또는 칸반뷰를 선택하면, LNB에서 공고를 변경하거나 페이지를 재진입해도 마지막으로 설정한 뷰 상태가 유지된다. 단, 공고 탭과 인재풀 탭은 각각 별도의 뷰 설정값을 저장한다.
- **검색어 유지:** 후보자 검색 후 노출 뷰(리스트/칸반)를 변경하더라도 입력된 검색 키워드는 초기화되지 않고 유지된다.
- **LNB 상태:** LNB의 접기/펼치기 상태는 세션 동안 유지되어야 하며, 마우스 호버 시 강조 효과와 툴팁이 제공된다.
- **데이터 초기화:**
  - 검색창의 'X' 버튼 선택 시 입력된 텍스트가 즉시 삭제된다.
  - 다중 선택 모드에서 'X' 버튼 또는 '선택 취소' 클릭 시 모든 체크박스 선택이 해제되고 플로팅 바가 비노출된다.
- **메일 수신 토글:** ON/OFF 변경 시 즉시 상태가 반영되며, 설정 시점 이후 발생하는 신규 후보자부터 적용된다.

# 3. 엣지 케이스 목록

- **데이터 부재 상태:**
  - 등록된 공고가 없는 경우: "등록된 공고가 없습니다" 문구 노출.
  - 검색 결과가 없는 경우: "검색결과가 없습니다" 문구 노출.
  - 후보자가 없는 상태에서 저장/인쇄 시도: "등록된 후보자가 없어요" 토스트 노출 (DEFECT-792).
- **데이터 제한 및 임계값:**
  - 검색어 제한: 2자 미만 입력 시 "2글자 이상 입력해주세요" 토스트 노출 (DEFECT-708).
  - 일괄 처리 제한: 저장/인쇄 시 1,000명 초과 시 "한 번에 처리 가능한 후보자는 최대 1,000명이에요" 토스트 노출 (DEFECT-797).
  - 명칭 초과: 공고명, 모집분야명, 후보자명이 영역을 벗어날 경우 말줄임(...) 처리.
- **시간 경과 제한 (90일 정책):**
  - 공고 마감 후 90일 경과 시: 채용 설정, 관리자 설정, 보고서, 발표 관리, 다중 선택, 후보자 등록, 저장/인쇄 등 주요 기능 클릭 시 "공고 마감일로부터 90일이 경과되어 이력서 열람 및 기능 이용이 불가 합니다" 토스트 노출.
  - 90일 경과 후보자: 다중 선택 체크박스 노출 제외 (DEFECT-824).

# 6. 중요도 S 항목

- **칸반뷰 노출 및 설정 확인:** 후보자 관리 진입 시 또는 뷰 설정 변경 시 칸반뷰가 정상적으로 노출되어야 하며, 노출뷰 설정 메뉴에서 '칸반'이 선택된 상태로 동기화되어야 한다.

- **공고 및 인재풀 부재 시 처리:** 등록된 공고와 인재풀 포지션 리스트가 모두 0건인 경우, 리스트 화면 대신 '후보자 관리 소개 화면'이 반드시 노출되어야 한다.

- **다중 선택 플로팅 영역 노출:** 칸반뷰에서 후보자를 1명 이상 선택할 경우, 화면 상단에 다중 액션을 수행할 수 있는 플로팅 바 영역이 즉시 노출되어야 한다. 해당 영역은 화면 스크롤 시에도 상단에 고정(Floating)되어야 한다.

#### 장기 — 멀티 에이전트 자동화

서비스 정책 컨텍스트를 각 에이전트가 공유하는 서비스 지식 베이스화.<br/>
각 에이전트는 독립적으로 동작하지만, 동일한 정책 컨텍스트를 공유하여 서비스 일관성을 유지한다.

```
서비스 정책 컨텍스트 (hiring knowledge base)
    │
    ├── 기획 에이전트    → 기존 정책 기반 신규 기능 기획서 검토
    ├── 디자인 에이전트  → UI 변경이 정책 플로우와 충돌하는지 검증
    ├── FE 에이전트      → diff 기반 영향도 분석 + 엣지 케이스 발굴 (현재)
    ├── BE 에이전트      → API 변경이 프론트 정책에 미치는 영향 분석
    ├── QA 에이전트      → 정책 기반 TC 자동 생성 + 결함 이력 참조
    ├── 배포 에이전트    → 고위험 변경 감지 시 배포 게이트 강화
    └── 모니터링 에이전트 → 정책 위반 패턴 실시간 감지
```

### NOTE

보완한 프롬프트에 따라 verify report 형식이 아래와 같이 직접 컴포넌트를 명시하는 형식으로 달라지기도 함 - 더 테스트해보고 싶은데 회사에서 제공하는 토큰이 부족하다...ㅜㅜ

```
*   ✅ **채용 설정 관리 플로우** — `HiringProcessMenu`에서 '평가표 설정' 등 관리 기능을 수행함.
*   ✅ **채용 관리자 및 보고서 확인 플로우** — `useRecruitMenuItems`에서 `recruitManager` 권한에 따라 메뉴 노출을 제어함.
*   ⚠️ **발표 관리 플로우** — 정책에는 사람인 공고에 한해 새창 이동한다고 되어 있으나, 제공된 코드 내에는 해당 이동 로직(window.open 등)이 명시되어 있지 않음.
*   ✅ **후보자 노출 뷰 전환 플로우** — `ApplicantsManageView`에서 `currentTab` 상태에 따라 `KanbanView`와 `ListView`를 조건부 렌더링함.
*   ✅ **후보자 검색 및 필터 적용 플로우** — `useRecruitMenuItems`에서 `cell`, `email` 정보를 다루며 검색 및 필터링을 위한 데이터 구조를 갖춤.
*   ✅ **추천인재 확인 플로우** — `ApplicantsManageView` 및 관련 툴바에서 추천인재 관련 로직을 처리함.
```
